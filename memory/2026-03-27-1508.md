## 🆚 vLLM vs llama.cpp: Key Differences

### 🎯 Primary Design Philosophy

**llama.cpp:**
- **Minimalist:** Plain C/C++, no dependencies
- **Portability:** Runs on CPU, GPU, Apple Silicon, even Raspberry Pi
- **Use case:** Local inference, testing, edge deployment
- **Philosophy:** "Run any model anywhere with minimal setup"

**vLLM:**
- **Performance-first:** Optimized for high-throughput serving
- **Batching:** Continuous batching of concurrent requests
- **Use case:** Production serving, multi-user workloads
- **Philosophy:** "Maximize throughput and efficiency at scale"

---

### 🧠 Memory Management

**llama.cpp:**
- Standard KV cache storage
- Simple allocation/deallocation
- Limited optimization for long contexts
- **Context scaling:** Linear memory growth with context length

**vLLM:**
- **PagedAttention** (paper: Kwon et al., SOSP 2023)
- **Virtual memory** for KV cache (like OS virtual memory)
- **Blocks** allocated on-demand, shared across requests
- **Prefix caching:** Reuse KV blocks for repeated prompts
- **Context scaling:** Sub-linear memory growth

---

### ⚡ Throughput & Batching

**llama.cpp:**
- **Single-stream:** Processes one request at a time (typically)
- **Batching:** Limited batch support in llama-server
- **Throughput:** ~30-100 tok/s per GPU (depends on model)
- **Latency:** Lower latency for single requests

**vLLM:**
- **Continuous batching:** Dynamically batches incoming requests
- **Throughput:** 2-5x higher than llama.cpp for serving
- **Latency:** Slightly higher for single requests, but better aggregate throughput
- **Concurrency:** Native support for hundreds of concurrent requests

---

### 🔧 Hardware Optimization

**llama.cpp:**
- **Broad support:** CPU, CUDA, Vulkan, SYCL, Metal, HIP
- **Fallback:** Works on anything (CPU-only if needed)
- **Optimization:** Good, but conservative for portability

**vLLM:**
- **GPU-focused:** Primarily CUDA (NVIDIA), with some ROCm support
- **Optimization:** Aggressive, cutting-edge kernels
- **FlashAttention:** Native support for FlashAttention v1/v2
- **Custom kernels:** PagedAttention, chunked prefill, etc.

---

### 📦 Model Support & Quantization

**llama.cpp:**
- **GGUF format:** Standard, widely adopted
- **Quantization:** Q4_K_M, Q5_K_M, Q8_0, etc.
- **Flexibility:** Easy to switch quantization levels
- **Models:** 100+ supported (Llama, Mistral, Qwen, etc.)

**vLLM:**
- **Multiple formats:** GGUF, PT, AWQ, GPTQ, FP8
- **Quantization:** AWQ, GPTQ, FP8 (native), INT4/8
- **Integration:** Direct HuggingFace model loading
- **Models:** 100+ supported (similar to llama.cpp)

---

### 🚀 API & Deployment

**llama.cpp:**
- **llama-server:** REST API (OpenAI-compatible)
- **CLI:** `llama-cli` for local testing
- **Deployment:** Simple, single-binary
- **Monitoring:** Basic metrics

**vLLM:**
- **OpenAI-compatible API:** Drop-in replacement for OpenAI
- **Async API:** Native async support
- **Deployment:** Docker, Kubernetes, Ray
- **Metrics:** Comprehensive Prometheus metrics
- **Management:** Model swapping, multi-model serving

### 🧠 Why Cache Matters More Than Bandwidth

LLM inference has this pattern:

1. **Prefill phase:** Process entire prompt (memory bandwidth matters)
2. **Decode phase:** Generate one token at a time (latency matters)

**Decode phase is where 3D V-Cache wins:**
- Each token requires loading model weights
- With 128MB L3 cache, **most weights stay in cache**
- Without cache: **DDR5 fetch** (100+ cycles latency)
- With cache: **L3 fetch** (15-20 cycles latency)

**Result:** 3D V-Cache gives **~5x faster** weight access during decode.
