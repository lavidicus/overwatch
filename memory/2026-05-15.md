## Context Bloat Discovery & Fix (19:25-19:34 UTC)
- Jeremy noticed node1 server logs showing 18.29 tok/s — much slower than my 53 tok/s API benchmark
- **Root cause (per Jeremy's correction):** Slot had grown to ~76K live context. KV cache was already computed, but attending over 76K tokens slowed decode. Not re-processing — the slowdown is from attention cost growing with context length.
- API benchmark only measured generation time for new tokens, not the full context attending cost
- Server-side logs are the real source of truth
- **Fix:** Restarted llama-server to clear slot context
- **Post-restart:** Back to 53.7 tok/s
- **Diagnostics run:**
  - `curl -s http://node1:11434/slots` — slot ID 0, 262144 context, currently clean (149 tokens max)
  - `curl -s http://node1:11434/metrics` — prompt_tokens_seconds: 206.9, predicted_tokens_seconds: 53.7
- **Key metrics for future debugging:**
  - `llamacpp:prompt_tokens_total` / `llamacpp:prompt_seconds_total` — prompt processing
  - `llamacpp:predicted_tokens_seconds` — generation speed
  - `llamacpp:n_tokens_max` — largest observed context
- **Jeremy's recommended config order:**
  - `-np 1 --cache-prompt --kv-unified --cache-ram 8192 --cache-idle-slots --ctx-checkpoints 32 --checkpoint-every-n-tokens 8192`
  - If checkpoint invalidation is the issue: `--no-cache-prompt` (loses prefix reuse but avoids reprocessing storms)
  - `curl -s http://node1:11434/slots/0?action=erase` — clear slot without full restart
- Jeremy moved the second P6000 from node2 to node1
- node1 now has 2× P6000 (~53 tok/s), node2 has 1× P6000 (~43 tok/s)
- Everything working fine — expected performance difference
- **Why:** Model is GPU-bound — decode speed is intrinsic to 35B model on Ampere GPU. Tuning knobs improve consistency/latency, not raw throughput.
