### Findings

**Status: All three Phase 2 requirements are COMPLETE and WORKING.**

#### 1. Provider Auto-Discovery ✅
- `POST /api/providers/:id/discover` — returns raw model data from provider API
- `POST /api/providers/:id/discover-all` — discovers AND auto-registers as ProviderModel entries
- Both handle vLLM/Ollama/llama.cpp (`/v1/models`), Ollama (`/api/tags`), OpenAI (`api.openai.com`)
- Smart parsing: extracts parameters, quantization, and size from model names
- Frontend: "Discover All" button on connected provider cards
- "Create & Auto-Detect Models" button in the Add Provider dialog

#### 2. Models Require Systems Setup ✅
- Systems page fully functional (CRUD + test connection + WhichLLM + scan)
- Models page shows "Discover from System" button (disabled when no systems)
- `POST /api/systems/:id/models/scan` — scans directory for GGUF files
- `POST /api/systems/:id/models/scan-tree` — recursive filesystem browser
- Multi-step flow: Select System → Browse Files → Inspect GGUF → Register Model
- HuggingFace search + download to specific system + target folder
- `POST /api/models/hf-download` queues downloads
- `POST /api/models/register-from-inspection` creates model from inspection data

#### 3. Model File Introspection ✅
- `GET /api/models/inspect?path=...&systemId=...` — reads GGUF binary header
- Uses Python parser on remote system (heredoc approach avoids shell escaping)
- Local files: Python script parses binary header directly
- Extracts ALL standard GGUF fields: name, architecture, parameters, quantization, size, context length, embedding length, block count, etc.
- MMV/vision model detection: finds companion `.gguf` files with mmproj suffix
- Graceful fallback: filename-based detection when binary parsing fails
- Tested successfully: Qwen3.6-35B-A3B model inspected correctly
  - Architecture: qwen35moe
  - Parameters: 35B
  - Quantization: Q4_K
  - Size: 19.71 GB
  - Vision companion: mmproj-Qwen3.6-35B-A3B...f16.gguf detected
