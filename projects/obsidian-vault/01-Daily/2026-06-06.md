## 07:20 UTC - Overwatch GGUF Remote Inspection FIXED

Remote GGUF introspection now works end-to-end over SSH.

**Root causes (3 bugs):**
1. `bash -c` with heredocs + Python single quotes = quoting nightmare over SSH
2. execSync default 1MB maxBuffer exceeded by 17MB GGUF metadata (vocab arrays)
3. No SSH key fallback to ~/.ssh/id_rsa when encryptedKey was null in DB

**Fix:** Replaced runRemoteSSH with file-based approach: write command to temp file → SCP → execute via `bash <file>` → cleanup. Added maxBuffer: 50MB. Added SSH key fallback.

**Test results:**
- vllm: Architecture qwen35moe ✅, 733 tensors ✅, 54 KV ✅, 34.37 GB ✅
- VM111: SSH works ✅, test file path didn't exist (expected)

### GGUF Inspection Fix (10:45 UTC)
- Root cause: Python GGUF parser output was 10MB+ (tokenizer vocab data), exceeding Node's execSync maxBuffer and triggering ENOBUFS errors
- Fix 1: Filter Python parser to only extract essential KV keys (45 keys → ~40 essential), skip large values using `skip_value()` instead of storing them
- Fix 2: Switch from `execSync` to `spawn` (async) to avoid shell buffer limits
- Fix 3: Write Python script to temp file instead of inline (avoids shell escaping issues)
- Fix 4: mmproj companion detection now uses glob-style matching (`mmproj-*.gguf` and `*-mmproj.gguf`) instead of exact name matching — quantization suffixes differ between base model and mmproj
- Fix 5: Changed `const isVisionModel` to `let isVisionModel` in parseGGUFMetadata — const was preventing reassignment when mmproj was found
- Fix 6: Route handler now sets `isVisionModel=true` and `hasMmproj=true` independently of architecture name
